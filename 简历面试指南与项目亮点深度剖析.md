## 🌟 一、 简历上的“项目成就”该怎么写？

你可以把项目包装成以下语境（按需精简字数填充进简历）：

**项目名称**：基于多智能体图调度的儿科临床合规用药审计引擎 (VeriMind-Med)
**角色**：全栈开发 / 核心系统架构师
**技术栈**：FastAPI / LangGraph / LangChain / ChromaDB / React / Pydantic / 智谱&DeepSeek LLMs
**项目描述**：
基于权威学术成果重构落地的医疗垂直类 RAG（检索增强生成）系统。系统利用复杂的 Agent DAG 有向无环流转，针对由于大语言模型“内部常识越界”引发的高危临床儿科用药安全性问题（如超说明书用药、年龄禁忌、体重折算错误），首创了一套具备自激防守门控的医药规则核验工作站。
**核心业绩（亮点）**：
- **自适应的三粒度检索池**：搭建 128 / 512 / 1024 Token 三重粒度向量知识池，根据用户查询诉求（ DETAIL/CONCEPT等意图 ）动态挂载，兼顾了局部高精提取与长文本语境。并给文档挂载静态专业 $W_{authority}$ 权重系数以影响算法调优。
- **构建微服务化的 LangGraph 引擎**：以状态机概念彻底打通 Router分发、Retriever执行、Generator核对 和 Auditor断案四大节点的循环路网。
- **独创防幻觉 Trust-Score 量化门控**：深度复现实战论文算法模型，使用公式对 LLM 内部不可靠知识与输出强行干涉，对于医学不忠直回答强力实行拦截（拦截率100%）。
- **极客级质量保障护航**：沉淀 Trap-QA 盲测大纲并构建防流控的高并发断点回归脚本，提供学术指标展示图表化集成面板。

## 💡 二、 面试官高频拷问及“满分回答话术”

面试官在看到这个项目时，一般最感兴趣的是**“你怎么防止大模型瞎胡说（幻觉）的？”**和**“多 Agent 框架你怎么保证稳定流转的？”**。

### 🚨 拷问 1：大模型在回答医疗问题时经常出错（发生幻觉），你这个系统是怎么解决医疗严谨性这个痛点的？
**✅ 你的神仙答复：**
“其实幻觉不光是胡说八道的问题，很多大模型在处理儿科用药时，**它最大的毛病是‘好心干坏事’**。我们的知识库可能没有这条规则，但模型凭着自己的通用常识硬塞了结论，这对临床系统是极其危险的。
我是这么解决的：
1. **先在 Prompt 层面加防弹衣**：我死锁了 Generator 的生成提示词，强制要求它‘严禁出现不在召回范围内的内容’，并添加了只有 150 字的极简字数阈值来控制它的发散思维。
2. **在系统级加裁判法庭机制（最核心）**：我的 LangGraph 里特意剥离了一个独立的 `Auditor` 节点。我不再寄希望于生成器一次生成对，而是让打分模型把‘召回相关性’评分和‘生成内容忠实度’评分加权计算（这就是简历上的 Trust-Score 公式）。只要裁判检查出生成的内容哪怕有一丝没在参考知识库里，哪怕医学上碰巧对了，我也让它忠实度直接判 0 分，走 `REJECTED` 拒绝回答。我这是‘宁可漏过，不可错杀’的安全底线。”

### 🚨 拷问 2：LangGraph 比直接调一个带 Prompt 的大模型好在哪？开发中遇到了什么坑？
**✅ 你的神仙答复：**
“直接调 LLM 或者是搞 AutoGen 这种自由聊天形式的智能体太失控了。LangGraph 的好处是它是**图结构（DAG）**，我可以明确定义状态在每个节点该变成什么样。相当于我在这家公司建了迎宾、档案员、文书、质监局，他们靠一本叫 `AuditState` 的共享全局账本传递工作流。
**我踩过的最深的坑就是异步引擎的状态写入覆盖（时序冲突问题）**。
前期我在 `router_node` 这些地方全写了 `async def`，结果发现在高度并发跑批量脚本测试时，LangGraph 传递给下家的 `intent` 枚举值和 `normalized_query` 总是串位或者在 Pydantic 初始化时转档失败（变成了字符串形式导致后端报错）。
*如何解决的*：我立马把单纯靠 prompt 处理结构化 JSON 的任务锁成了**强类型化的 LangChain `with_structured_output`** 包装方法；其次，对于不需要漫长网络 I/O 而只是轻微做状态分发判决的 Router 节点和 Auditor 节点，我强行砍掉了它的 async，把它降级为标准的同步阻塞函数，彻底杜绝了并发流转时事件循环（Event Loop）的状态指针脏写问题。”

### 🚨 拷问 3：你在调用智谱这种外部大语言模型接口进行数据大批次检验时，有没有崩过？
**✅ 你的神仙答复：**
“非常有意思，崩过！而且崩得有点喜剧色彩。
当我在跑 10 道题的 Trap-QA 对抗测试时，我给我们的生成节点上了‘限制回复在 150 字以内，极致精简’的规则。结果大模型回复太快了，1 题跑完只要 1 秒，直接导致后边的 `for` 循环变成了对智谱免费 API 的高并发轰炸。
然后果不其然，接口传回了 `429 Too Many Requests` （触发限流防洪），后边的题直接全变成了 ERROR。
*如何解决的*：我马上重做了我的压力测试脚本 `run_trap_qa.py`：第一，实现了**断点续传**（直接通过对 list 切片跳过前面做完的题开始跑）；第二，极其朴素却管用地增加了一段 `asyncio.sleep(5)` 休眠队列代码。最后项目终于平滑把所有对抗题的评测单跑穿，完美展示！”

### 🚨 拷问 4：为什么要搞 128 / 512 / 1024 Token 这三个知识库粒度？这难道不占空间吗？
**✅ 你的神仙答复：**
“这是医学长句容错性的必须手段。我在调研中发现：
如果只切小块（如 128 Token），它非常适合捞出诸如‘布洛芬儿童 5-10mg/kg’这种精确剂量，但是如果是大段的病理表述，上下文明明有关联却从中间被生生切断断连了；如果只切大板块（1024），找精准的数字又容易被噪声稀释。
所以我在 ChromaDB 里建了三个不同维度的合集，并在我的 LangGraph Router（路由节点）设定了：只要提问是要求找特定剂量（DETAIL），就自动映射去调 128库；只要是在问适应情况或机制（CONCEPT/CONTEXT），就去找 512 或者 1024 库。
相当于这不叫占空间冗余，这叫‘长短镜头的双盲自适应切换’。”
