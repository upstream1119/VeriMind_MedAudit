# CRAIC V1 设计文档（儿科用药审计项目）

## 1. 背景与约束
- 目标：构建可运行、可演示、可解释的儿科用药审计系统。
- 医生反馈的最高风险错误集中在两类：
  1. 药物联用（尤其与抗生素联用时）
  2. 使用剂量（剂量大小、单位、频次、体重折算）
- 时间约束：
  - V1 截止：2026-05-02
  - 校赛窗口：预计 2026-05-15 左右
  - 期中考试：2026-04-22 / 2026-04-23（主开发从 2026-04-24 开始）

## 2. 目标与非目标
### 2.1 V1 目标（2026-05-02）
1. 可运行：后端审计链路与前端展示链路完整可用。
2. 可演示：仅预置题稳定演示，不依赖自由提问。
3. 可验证：预置题通过率 `>= 90%`。
4. 可解释：结果展示证据链与 Trust-Score 分项。

### 2.2 非目标（V1 不做）
1. 不追求开放域问答稳定性。
2. 不在 V1 并行扩展过多新能力。
3. 不做与比赛主链路无关的大规模重构。

## 3. 优先级与资源
### 3.1 总体优先级
- 安全 / 演示 / 评测 = `50 / 30 / 20`

### 3.2 校赛冲刺权重
- 稳定性与观感 / 新能力 = `65 / 35`

### 3.3 V1 可投入预算
- 约 18 ~ 27 小时（按 2026-04-24 到 2026-05-02，每天 2~3 小时估算）

## 4. 方案对比与选型
### 方案 A：安全闭环优先 + 双主演示（采用）
- 先封住高危错误（联用 / 剂量），再清楚展示完整审计链。
- 优点：稳定、可讲、比赛风险最低。
- 风险：V1 阶段新增功能数量受控。

### 方案 B：观感与能力并进
- 优点：看起来更“全面”。
- 风险：在当前工时下容易两头不到岸。

### 方案 C：能力扩展优先
- 优点：功能点多。
- 风险：稳定性下降，不利于现场演示。

## 5. V1 系统设计
### 5.1 演示主链路
1. 输入预置高危题。
2. 路由模块完成意图识别与问题标准化。
3. 检索模块召回证据并返回权重信息。
4. 生成模块在证据约束下生成草稿回答。
5. 审计模块计算分项分数并执行三态门控（TRUSTED / WARNING / REJECTED）。
6. 前端展示：结论、证据、分项分数。

### 5.2 高危场景覆盖
1. 联用风险：优先覆盖“抗生素联用”冲突与禁忌场景。
2. 剂量风险：优先覆盖“单位、频次、体重折算”三类错法。

### 5.3 前端演示要求
1. 第一屏：高危拦截结果（是否拒绝 / 警告）。
2. 第二屏：证据链与分项评分（支持现场讲解）。
3. 固定演示脚本：固定题目、固定顺序、固定口径。

## 6. 验收标准
### 6.1 定量指标
- 预置题通过率 `>= 90%`

### 6.2 定性指标
1. 全链路演示不依赖临时手工修补。
2. 关键字段可解释：`intent / evidence / trust_score / trust_level`。
3. 结果报告同时给出两类指标并明确区分：
   - 通过门控后的幻觉率（passed-hallucination）
   - 覆盖率（coverage）

## 7. 风险与缓解
1. 范围膨胀风险：冻结 V1 范围，仅做预置题稳定演示。
2. 接口不稳定风险：准备非流式兜底演示路径。
3. 现场变量风险：自由提问不作为主流程，仅作备选。

## 8. 面试叙事设计
每个关键改动都按同一模板沉淀：
1. 问题现象
2. 根因定位
3. 修复策略
4. 验证证据

目标：形成 3~5 个可复述的技术故事，覆盖“安全性、工程实现、调试能力”。
