# VeriMind-Med 阶段与任务流程规范

## 1. 原则与思想
- **渐进式交付 (Incremental Delivery)**：复杂的 AI 工程无法一蹴而就。整个项目必须由几个边界清晰的阶段共同组成，上个阶段未稳定前绝不贸然开启下一个。
- **原子任务追踪**：阶段内需包含一系列高内聚的小颗粒度任务（Checklist）。一项完成才能打钩，并在阶段完成后产出《总结文档》。
- **隔离验证环境**：每一个模块（如：前端、LLM API 接口、向量存储）在串联前均需要独立的回归验证标准。

## 2. 标准工程流程 (SOP)

### Step 1: 制定 / 更新全景图 ( implementation_plan.md & task.md )
- **前置**：在每次着手新阶段或接手陌生系统前，必须首先阅读所有依赖项和理论背景（如：相关论文、参考仓库等），然后撰写或更新实施计划与 Checklists。
- **动作**：在 `task.md` 文件中按子任务定义好 TODO 列表，标记所有项为 `[ ]`。

### Step 2: 渐进式实施 & 原型快速验证
- 严格遵循 `task.md` 的路线和次序。开始一项任务将 `[ ]` 变更为 `[/]`。
- 采用测试驱动风格。后端每写完一个 Router 或数据 Model，随即利用 curl/requests 构建联调请求；前端每封装完一个 API 调用即在 Local State 中 Mock 验证数据视图。
- 任务完结后更新为 `[x]`。

### Step 3: 模块联调 & Bug 阻断
- 定向测试：如果在接驳新模型 API 时出现 50x 或连通性断裂问题，必须回到纯单体脚本测试该服务接口，严禁掩耳盗铃直接大面积换用其他的实现方法。

### Step 4: 知识固化 & 阶段总结
- 阶段内所有任务清单 `[x]` 并跑通系统联调后，项目执行中断。
- **固化文档**：必须在 `阶段性工作总结/` 文件夹下生成一份格式清晰的 `.md` Markdown 文件。
- **汇报**：向管理者与后继实施人员如实记载：完成了哪些接口、核心配置有何改变（如有无新补充的重要参数项）、以及下一步执行重点。
